Case 01核心案例

植物工厂 MES 系统开发

把自动化生产现场的分散信息，转化为可追踪、可迭代的软件需求。

+
周期2024.04 - 至今
角色需求负责人 / 项目推进
对象终端用户、研发团队
领域植物工厂数字化生产

01 业务背景与问题

植物工厂生产包含种植任务、设备动作、人员操作和现场异常。依赖人工记录时，信息容易分散，问题定位与过程追溯成本较高。项目目标是用 MES 承接自动化生产业务，并让系统真正适配现场。

02 我的职责

  • 对接终端使用方，收集实际操作反馈和新增需求。
  • 将现场语言整理为研发可执行的功能与流程要求。
  • 维护需求清单，跟踪开发、调试、反馈和后续优化。

03 推进过程

  1. 按生产流程拆解角色、任务、状态与异常节点。
  2. 区分核心流程、体验优化与现场适配问题，明确优先级。
  3. 与研发同步业务规则，跟进实现情况并组织使用方验证。
  4. 把落地反馈重新纳入需求池，形成持续迭代闭环。

04 交付与价值

  • 形成面向自动化生产业务的软件需求框架。
  • 建立终端使用方与研发团队之间的稳定反馈通道。
  • 持续推动产品迭代、现场调试与实际业务适配。

自评与下一轮优化

优势：能用农业生产经验识别真实操作约束，并持续跟进到落地。不足：履历中尚缺少上线前后效率、错误率等量化对比。改进：后续建立“需求来源—优先级—版本—验收指标”台账，并补充工时、操作步骤和异常闭环时长等基线数据。

Case 02产品落地

农业智能控制器项目

从需求调研、PRD 和交互原型，到第一代产品量产与第二代迭代。

+
周期2022.07 - 至今
角色产品项目协同
工具PRD、墨刀、Easybuilder
结果第一代落地量产

01 产品挑战

农业控制器连接传感器、执行设备与现场人员，既要满足控制逻辑，也要降低非专业用户的操作门槛。产品工作需要同时协调 UI、软件、硬件和 PLC 侧限制。

02 我的职责

  • 开展需求调研与功能梳理，撰写 PRD。
  • 使用墨刀制作交互原型，先验证信息架构和操作路径。
  • 对接 UI、软硬件团队，跟进开发问题和版本进度。
  • 学习 Easybuilder，参与 HMI 页面配置和流程优化。

03 关键做法

  1. 围绕设备状态、参数设置、手动控制和异常提示组织功能。
  2. 把复杂参数转化为清晰的页面层级与操作反馈。
  3. 在原型阶段暴露流程冲突，降低进入开发后的反复沟通。
  4. 根据一代落地经验推进二代原型迭代。

04 交付与价值

  • 完成需求文档与多轮交互原型输出。
  • 协同推动第一代智能控制器落地量产。
  • 形成第二代产品原型与持续优化方向。
  • 建立从软件交互到 HMI 控制界面的跨端理解。

自评与下一轮优化

优势：学习速度快，能跨越农业业务、交互设计和硬件控制边界。不足：当前案例缺少用户访谈样本量、版本节点和量产后的使用反馈。改进：补建版本路线图，记录关键决策依据，并用任务成功率、误操作次数和完成时长验证二代交互。

Case 03质量保障

PAR 传感器配套 APP 测试

用 61 条测试用例和 3 轮回归测试，覆盖核心功能与异常场景。

+
周期2024 - 2025
角色产品定义 / 测试执行
测试用例61 条
回归轮次3 轮

01 测试目标

PAR 传感器面向农业科研场景，配套 APP 不仅要正确展示数据，还要处理设备连接、状态变化、输入边界和异常操作，避免界面问题影响测量使用。

02 用例设计

  • 按核心功能、界面交互、数据展示和异常场景拆分测试范围。
  • 明确前置条件、操作步骤、预期结果和实际结果。
  • 对发现的问题进行复现描述，便于研发定位。

03 闭环过程

  1. 完成 61 条全量测试用例并执行首轮验证。
  2. 汇总 UI 与功能缺陷，同步研发修复。
  3. 针对修复项及关联功能开展 3 轮回归。
  4. 持续确认问题状态，避免修复引入新的影响。

04 交付与价值

  • 形成结构化测试表与问题反馈记录。
  • 推动修复多项 UI 和功能缺陷。
  • 通过多轮回归提升版本交付前的稳定性。
  • 把农业使用场景纳入软件质量验证。

自评与下一轮优化

优势：测试执行完整，能够从科研用户和农业场景识别边界问题。不足：缺陷总量、严重程度分布与最终关闭率尚未沉淀。改进：引入缺陷等级、模块覆盖率、通过率和遗留风险清单，让测试结论更可衡量。

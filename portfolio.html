代表工作样本

重点不是项目名称，而是这些样本如何证明你能把问题推进成结果。

需求拆解 / 规则定义 / 开发协同

资源与工单时间调整规则样本

体现复杂规则拆解、原型表达、开发协同和交付闭环能力。

生产现场 MES 场景中，工单提前、资源占用、冲突提醒和中途取消会互相影响，口头描述很容易造成开发理解偏差。

产出物

规则需求说明、开发理解稿、状态流程图、冲突判断图、可点击交互原型。

关键判断

这类需求不能只写页面按钮，而要先拆清状态、时间、资源位、异常和人工确认边界，再进入原型与开发说明。

协作方式

将业务规则翻译成研发可实现、测试可验证的材料，并通过流程图和原型降低沟通成本。

产品定义 / 原型推进

智能控制器产品样本

体现产品定义、原型表达和软硬件协同能力。

控制器产品从 0 到 1 过程中，需求、界面、硬件和使用场景需要反复磨合。

产出物

PRD、3轮交互原型验证、HMI 页面逻辑说明、迭代反馈记录。

关键判断

先把用户操作路径和页面反馈做顺，再交给硬件和软件细化实现，能显著降低反复沟通。

协作方式

设计、硬件、软件共同参与迭代，产品侧负责把信息翻译成统一语言。

测试设计 / 质量收口

APP 测试闭环样本

体现结构化测试思维和交付收口能力。

配套 APP 在交付前需要把主流程、异常场景和界面细节都验证到位。

产出物

61 项测试场景、3轮回归记录、问题清单和修复确认结果。

关键判断

测试价值不只是找 bug，而是建立一套交付前可验证的质量标准。

协作方式

与研发反复确认问题复现路径、修复结果和版本风险点。

客户对接 / 执行闭环

订单交付执行样本

体现执行力、对外沟通和全流程交付能力。

园艺礼品订单需要从客户需求确认一路跟到生产、发货和验收。

产出物

需求确认、执行跟进、异常协调、交付闭环记录。

关键判断

订单不是事务处理，而是对需求边界、节奏和风险的持续管理。

协作方式

客户、生产、供应环节之间多线沟通，减少信息偏差和返工。
